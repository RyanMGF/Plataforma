Introdução ao Git e GitHub

O que é isso? Um "save game" para seus projetos!

Imagine que você está criando um projeto (um texto, um site, um jogo). A cada mudança, você fica com medo de estragar algo, certo? Git é como um sistema de "save" superpoderoso no seu computador, que te deixa salvar "checkpoints" do seu trabalho. O GitHub é como uma "nuvem" online onde você guarda esses saves e pode compartilhar com amigos para trabalharem juntos.

Git vs. GitHub: Qual a Diferença?

Este é o ponto que mais confunde iniciantes! É crucial entender a diferença:

  • Git: É o programa que você instala no seu computador. Ele é a ferramenta que controla as versões e o histórico. Ele funciona 100% offline, sem precisar de internet.
  • GitHub: É um site (uma plataforma na internet). Ele é um serviço que hospeda (guarda) seus projetos que usam Git. É a "rede social" do seu código, permitindo backup e colaboração.

Analogia Final: Pense no Git como o Microsoft Word, o programa que você usa para escrever um documento. Pense no GitHub como o Google Drive, o serviço online onde você salva e compartilha seus documentos do Word.

1. Git: A máquina do tempo no seu computador

O Git funciona localmente, ou seja, só na sua máquina. Ele cria um histórico de todas as alterações que você faz nos seus arquivos. Se algo der errado, você pode simplesmente "voltar no tempo" para uma versão anterior que estava funcionando!

Analogia: Pense no Git como o botão "Salvar" de um jogo. Cada vez que você salva, cria um ponto seguro. Se você perder a fase, pode voltar para o último save. O Git faz isso com o seu código.

Os 3 Comandos Mágicos para "Salvar o Jogo"

Para criar um "checkpoint" (que chamamos de commit), você segue 3 passos:

Passo 1: `git add` - Separar o que vai ser salvo

Este comando escolhe quais arquivos modificados você quer incluir no próximo save. Ele os move para um lugar chamado "Staging Area" (Área de Preparação).

Mas por que esse passo extra? A Staging Area permite que você crie "saves" (commits) muito organizados. Imagine que você modificou 3 arquivos, mas apenas 2 deles estão relacionados a uma nova funcionalidade. Você pode usar `git add` apenas nesses 2 arquivos para criar um commit limpo e temático.

-- Para adicionar um arquivo específico:
git add nome_do_arquivo.txt

-- Para adicionar TODOS os arquivos modificados de uma vez (use com cuidado!):
git add .

Atividade Prática: Preparando um arquivo

Abaixo, simulamos que você tem um arquivo modificado. Clique no botão para simular o comando git add index.html e veja o que acontece.

Status do Git:
  - index.html (modificado, não preparado)

Passo 2: `git commit` - Criar o "save" com uma descrição

Este é o comando que realmente cria o checkpoint. A parte -m serve para você deixar uma mensagem explicando o que você mudou. Boas mensagens são super importantes!

-- 'Initial commit' é uma mensagem comum para o primeiro save.
git commit -m "Adicionei o título da página inicial"

Passo 3: `git log` - Ver seu histórico de saves

Quer ver todos os checkpoints que você já criou? Use este comando.

git log

A saída será algo parecido com isto, mostrando cada commit, seu autor e a mensagem:

commit a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0 (HEAD -> main)
Author: Seu Nome 
Date:   Mon Oct 26 20:00:00 2023 -0300

    Adiciona o título da página inicial

2. GitHub: A rede social dos seus projetos

O GitHub é uma plataforma online que "hospeda" seus projetos que usam Git. Ele é o lugar onde a mágica do trabalho em equipe acontece. Além de ser um backup seguro, é lá que você pode mostrar seus projetos para o mundo!

Analogia: Se o Git é o seu console de videogame, o GitHub é a rede online (como a PSN ou a Xbox Live) onde você pode guardar seus saves na nuvem e convidar amigos para jogar no mesmo mundo que você.

Comandos para conversar com o GitHub

Para conectar seu projeto local (Git) com o projeto online (GitHub), usamos alguns comandos chave:

`git clone` - Baixar um projeto pela primeira vez

Viu um projeto legal no GitHub e quer uma cópia dele no seu computador? Você "clona" o projeto.

git clone https://github.com/usuario/nome-do-projeto.git

`git push` - Enviar seus "saves" para a nuvem

Depois de fazer alguns commits na sua máquina, você usa o push para enviar todas essas novidades para o GitHub. É como fazer o "upload" dos seus saves.

-- 'origin' é o nome padrão do seu repositório no GitHub.
-- 'main' é o nome padrão da "linha do tempo" principal do seu projeto.
git push origin main

`git pull` - Baixar as novidades da nuvem

Se um amigo seu fez alguma alteração no projeto e enviou para o GitHub, você usa o pull para baixar e atualizar o seu projeto local. É como fazer o "download" das atualizações.

git pull origin main

Desafio: Seu primeiro fluxo de trabalho!

Você acabou de criar um arquivo HTML novo para o seu projeto e quer salvar essa alteração e enviá-la para o GitHub. Qual seria a sequência correta de comandos?

Sua missão: Ordene os 3 comandos que você usaria para completar a tarefa.

A sequência correta seria:

  1. git add . (ou git add nome_do_arquivo.html) para colocar o novo arquivo na "caixa de preparação".
  2. git commit -m "Cria arquivo HTML inicial" para criar o "save" local com uma mensagem clara.
  3. git push origin main para enviar seu novo save para o GitHub.
